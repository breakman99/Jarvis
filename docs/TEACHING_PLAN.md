> 本文档用于记录你在 Jarvis 项目中的 Agent 学习规划与进度，后续可以随时调整和补充。

## 总体目标

- **目标 1**：吃透当前这个「最小可用工具调用 Agent」的实现细节，能自己从零写出来。
- **目标 2**：在 Jarvis 上逐步实现：更丰富的工具、思考能力、记忆系统、多工具编排。
- **目标 3**：把 Jarvis 打造成一个可以持续扩展、持续练习 Agent 技术的个人实验场。

## 第 1 阶段：吃透当前 Agent 实现

**阶段目标**：完全理解当前 `AgentApp` + `LLMGateway` + 工具调用的整体流程，能画图 & 口述。

### 需要掌握的关键点

- **消息与对话历史**
  - 理解 `messages = [{"role": "...", "content": "..."}]` 的结构。
  - 理解为什么要把每一轮的 user / assistant / tool 结果都 append 到 `messages` 中。
- **工具调用机制**
  - 理解 `tools` / `tool_choice` 的作用。
  - 理解模型返回的 `tool_calls` 结构（`id`、`name`、`arguments`）。
- **Agent 循环**
  - 理解当前 `SimpleAgent.run` 中的循环逻辑：
    - 向模型发起请求 → 检查是否有 `tool_calls` → 有则执行工具并把结果写回 → 再问模型。
  - 理解为什么要有 `max_iterations` 这样的上限，防止死循环。

### 建议练习任务

- [ ] **流程图练习**
  - 用你习惯的方式（纸笔 / Markdown / 工具）为 `SimpleAgent.run` 画一张简单流程图。
  - 能清晰标出：用户输入 → 模型回复 → 工具调用 → 工具执行 → 再次询问模型 → 最终响应。

- [ ] **多样化 Prompt 测试**
  - 只问时间：例如「现在几点了？」。
  - 只算加法：例如「帮我算 12.3 + 45.6」。
  - 同时问时间和加法。
  - 问一个明显不需要工具的问题（例如「给我讲个笑话」），观察模型是否直接回答。

- [ ] **观察内部状态**
  - 在不改变整体结构的前提下，适当 `print`：
    - 每一轮循环时的 `self.messages` 长什么样。
    - 每次 `resp_msg.tool_calls` 的内容。
  - 要求自己能**口述**：
    - 如果模型一次调用了多个工具，代码会如何处理？
    - 如果模型一直在调用工具直到超过上限，会发生什么？

## 第 2 阶段：构建更实用的工具层

**阶段目标**：从「演示用工具」升级为「能真正帮你完成一些小任务」的工具集。

### 需要掌握的关键点

- **工具 API 设计**
  - 工具的输入参数应该清晰可控（类型、范围）。
  - 工具的返回结果要尽量「对模型友好」，用自然语言或者结构化文本。
- **参数 Schema 与模型对齐**
  - 理解 `TOOLS_SCHEMA` 中 `description`、`parameters` 的作用。
  - 学会通过 `description` 引导模型在什么场景下使用该工具。

### 建议在 `tools.builtin` 中扩展的工具示例

- [ ] **文件工具（本地）**
  - 例如：读取某个固定目录下文本文件的内容（注意限制范围，避免太危险）。
- [ ] **网络工具（HTTP GET）**
  - 使用 `requests` 做简单的 GET 请求，返回简短的摘要或部分文本。
- [ ] **文本处理工具**
  - 例如：统计某个单词在文本中出现的次数。
  - 例如：从文本中提取邮箱 / URL 等。

### 错误处理练习

- [ ] 为工具增加基础错误处理：
  - 参数缺失 / 类型错误时的返回说明。
  - 网络失败或文件不存在时的错误提示。
- [ ] 思考并统一约定：
  - 当工具执行失败时，返回给模型的字符串格式是什么样最容易被理解和转述给用户。

## 第 3 阶段：升级为“思考型 Agent”

**阶段目标**：让 Agent 不只是「一问一答 + 工具」，而是能自己规划步骤、分步执行。

### 需要掌握的关键点

- **ReAct / Plan-and-Execute 思想**
  - 先思考计划，再决定要不要调用工具。
  - 区分「中间思考过程」和「最终回复给用户的话」。
- **在 Prompt 中约束思考方式**
  - 通过 system prompt 要求模型：
    - 先用中文列出解决步骤。
    - 再根据步骤决定是否调用工具。

### 建议练习任务

- [ ] 修改或扩展 `SimpleAgent` 的 system prompt
  - 让模型在回答前先给出「解决步骤」的文字描述。
- [ ] 设计一个需要多步推理的任务
  - 例如：「先查当前时间，根据是上午还是下午推荐一首歌名，然后用礼貌语气向我问好」。
- [ ] 尝试实现「思考多轮」能力
  - 即：即使没有工具调用，也允许模型通过多轮 assistant 自我对话，先产出计划再执行。

## 第 4 阶段：记忆与状态管理

**阶段目标**：让 Agent 能记住一些长期信息（例如用户偏好），而不是每次从零开始。

### 需要掌握的关键点

- **短期记忆 vs 长期记忆**
  - 短期：当前对话的 `messages`。
  - 长期：写到文件 / 数据库 / 向量库中的持久信息。
- **记忆写入与读取的时机**
  - 何时把信息写进长期记忆。
  - 什么时候把记忆内容注入到每次对话的开头。

### 建议实现的最小记忆系统

- [ ] 在项目中新增一个简单的长期记忆存储（如 `memory.json`）
  - 用于存储用户的称呼、常用语言、风格偏好等。
- [ ] 设计触发条件
  - 当用户说「我叫 XXX」「以后叫我 XXX」之类的话时，写入记忆。
- [ ] 在 Agent 初始化或每次对话前，自动注入记忆
  - 把关键记忆内容以 system / assistant 消息的形式加入 `messages`，让模型知道这些设定。

## 第 5 阶段：多工具编排与错误恢复

**阶段目标**：让 Agent 能够组合多个工具、在错误情况下做出合理的恢复或解释。

### 需要掌握的关键点

- **工具之间的数据流**
  - 一个工具的输出可以作为另一个工具的输入。
  - 在 `handle_tool_calls` 中管理好执行顺序和依赖。
- **错误检测与恢复策略**
  - 捕获异常并转化为模型易于理解的自然语言。
  - 让模型在工具失败时尝试解释原因或提供备选方案。

### 建议练习任务

- [ ] 设计一个需要多个工具配合的任务
  - 例如：「从配置文件中读取一个 URL → 访问该 URL → 摘要内容并回复给用户」。
- [ ] 为 `handle_tool_calls` 增加错误捕获
  - 使用 `try/except` 捕获工具执行时的异常。
  - 将错误以统一格式写入 `messages`，方便模型做后续解释。
- [ ] 在 system prompt 中约定错误处理策略
  - 遇到工具错误时，让模型：
    - 分析错误原因。
    - 向用户说明问题。
    - 如果可能，给出替代方案。

## 第 6 阶段（进阶）：多 Agent、插件化工具、对外 API

**阶段目标**：把 Jarvis 从个人实验脚本升级为可以扩展、可复用的小框架。

### 可能的方向（按兴趣选择）

- [ ] **多 Agent 协作**
  - 设计不同角色的 Agent：
    - 规划 Agent：负责拆解任务。
    - 执行 Agent：专门调用工具。
    - 审阅 Agent：检查结果、润色回答。
  - 用一个「管理器」来协调它们之间的信息流。

- [ ] **插件化工具系统**
  - 将每个工具作为模块放在 `tools/` 目录下。
  - 启动时自动扫描模块并注册到 `AVAILABLE_FUNCTIONS` 和 `TOOLS_SCHEMA` 中。

- [ ] **对外提供 API 服务**
  - 使用 FastAPI / Flask 将 Agent 封装成 HTTP 服务。
  - 暴露一个 `/chat` 接口，外部应用可以通过 HTTP 调用 Jarvis。
